Every bad prompt, every hallucination, every confusing agent behavior traces back to a misunderstanding of what an LLM is actually doing. This module gives you the mental models that will make every technique in this course feel obvious rather than magical.

1.1 The Autocomplete That Read the Entire Internet

An LLM does one thing: given a sequence of tokens, it predicts which token comes next. That's it. Everything you see — reasoning, code, stories, answers — emerges from doing this prediction billions of times with a model trained on most of the text humanity has ever written.

A token is roughly 3–4 characters. "Hello world" is 2 tokens. "supercalifragilistic" is 6. The model never sees letters — it sees integer IDs that map to these chunks. The vocabulary is fixed at training time (Claude uses ~100k tokens).

During training, the model is shown trillions of tokens and learns to predict each next token. It adjusts billions of internal weights until its predictions become accurate. After training, the weights are frozen — the model doesn't learn anything new from your conversations.

The model does not retrieve facts from a database. It compresses statistical patterns from training data into weights. This is why it can be confidently wrong — it's completing a pattern, not looking something up. This is also why RAG (Module 5) is so important. 
# A rough mental model of what happens at inference time:
input_tokens = tokenize("The capital of France is")
# model predicts: " Paris" with probability 0.97
# model predicts: " Lyon" with probability 0.01
# model predicts: " a" with probability 0.006
# ...
output = sample(predictions)  # usually "Paris"

The temperature setting controls how much randomness is added to this sampling. Temperature 0 = always pick the most likely token. Temperature 1 = sample proportionally. This is why "temperature 0" gives you deterministic outputs.

1.2 The Context Window Is Everything

The model has no persistent memory between conversations. Each call starts fresh. The only thing the model can "see" is what's inside the context window right now — and your entire job as a prompt or context engineer is to put the right information in that window.

Think of the context window as a shared whiteboard. You write on it, the model reads it, writes its response, and forgets everything when you close the session. The window has a fixed size — Claude's is up to 200k tokens. Once you hit the limit, old content is truncated. 
# Everything the model processes in a single API call:
{
  "system": "You are a helpful assistant...",   # system prompt
  "messages": [
    {"role": "user",      "content": "What is RAG?"},
    {"role": "assistant", "content": "RAG stands for..."},
    {"role": "user",      "content": "How do I implement it?"}
  ]
}
# The model sees ALL of this at once, left to right, and predicts the next token.

This means long conversations get expensive fast, and attention quality can degrade on very long inputs. You'll learn how to manage this deliberately in Module 4.

💡 The mental model that made this click: the context window is not "chat history" — it's the model's entire working memory. When you say "remember what I said earlier," the model doesn't remember — it reads the transcript. If the transcript doesn't include it, it's gone.
1.3 How Training Shapes Behavior (RLHF & System Prompts)

Raw pretraining produces a model that completes text. RLHF (Reinforcement Learning from Human Feedback) fine-tunes it to be helpful, harmless, and honest. This is why Claude follows instructions, refuses harmful requests, and maintains a consistent personality.

The system prompt is injected before every conversation, at the highest trust level. It's where you configure the model's persona, constraints, output format, and domain expertise. The user has less influence than the system prompt in Claude's trust hierarchy. 
# Three tiers of influence (highest to lowest):
1. Anthropic's training  →  baked into weights, can't be overridden
2. System prompt         →  set by you (the developer/operator)
3. User messages         →  your end-users at runtime

Understanding this hierarchy explains why "jailbreaks" mostly fail and why a well-written system prompt can dramatically change the model's behavior. You're configuring a system, not arguing with a person.

System prompts are not secret — a determined user can often extract them via prompt injection. Never put secrets, API keys, or truly sensitive business logic in a system prompt. Use it for persona, format, and safety constraints only.
