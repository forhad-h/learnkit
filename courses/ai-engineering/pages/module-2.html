Prompting is not about magic words — it's about giving the model the right information, role, and constraints so it can produce the output you actually need. Master these fundamentals and 80% of your prompt problems disappear.

2.1 The Four Parts of Every Good Prompt

Every prompt that works well contains some combination of four components. You won't always need all four, but knowing which is missing is how you debug a bad output.

ComponentWhat it doesExample
RoleSets expertise and voice"You are a senior Python engineer."
ContextBackground the model needs"This is a FastAPI app used by 10k users/day."
TaskWhat you want done"Refactor this function to be async."
FormatShape of the output"Return only the updated function, no explanation."
 
# Weak prompt:
"Fix my code."

# Strong prompt (all four components):
"""
You are a senior Python engineer (ROLE).
This function is called on every HTTP request in a high-traffic API (CONTEXT).
Refactor it to remove the synchronous database call and replace it with async/await (TASK).
Return only the updated function with no explanation or markdown fences (FORMAT).

[code here]
"""
The most commonly missing component is Format. If you don't specify the output shape, the model will pick one — usually verbose markdown with headers and bullet points. Always say what you want back.
2.2 System Prompts vs User Prompts — Use Both

The system prompt is your application's persistent configuration. The user prompt is per-request input. Mixing them together in one big user prompt works, but it wastes the cache and makes your prompts hard to maintain.

import anthropic

client = anthropic.Anthropic()

SYSTEM = """
You are a code review assistant for Python projects.
Always respond with:
1. A severity label: [CRITICAL] / [WARNING] / [INFO]
2. The specific line number(s) affected
3. A one-sentence explanation
4. A corrected code snippet

Never add introductions or closing remarks.
"""

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=SYSTEM,
    messages=[
        {"role": "user", "content": f"Review this function:\n\n{code}"}
    ]
)
Claude supports prompt caching: if you send the same system prompt repeatedly, Anthropic caches it and charges you much less per token. A 10,000-token system prompt costs 90% less on cache hits. Always put stable content in the system prompt and dynamic content in user messages.
2.3 Few-Shot Examples: Show, Don't Tell

If you can't get the format right by describing it, show the model an example. Two or three well-chosen examples (few-shot) are worth a page of instructions.

SYSTEM = """
You extract action items from meeting notes.
Output format — one item per line, no bullets, no numbering:

OWNER: ACTION (DEADLINE)

Examples:
Alice: Update API docs to include new auth endpoints (Friday)
Bob: Set up staging environment for QA team (Next Monday)
Unassigned: Review Q2 budget proposal (EOW)
"""

user_message = """
Meeting notes from 2026-04-22:
- John said he'll handle the onboarding emails by Thursday
- Sarah and the design team need to finalize the new dashboard mockups before the client call
- No one owns the security audit yet, needs to be done by end of month
"""
💡 The trick I use: write one perfect example output by hand, then use that as your few-shot example. If you struggle to write the example yourself, your format spec is too vague.
More examples is not always better. 2–3 examples that cover edge cases outperform 10 examples of the same pattern. Include at least one example that shows what to do when the input is ambiguous or missing information.
2.4 Output Constraints: JSON, XML, and Structured Data

For programmatic use, you need reliable structured output. Asking nicely doesn't always work — use a JSON schema, prefill the assistant's response, or use the API's structured output features.

import json

SYSTEM = """
You extract product information from user messages.
Always respond with valid JSON matching this schema exactly:

{
  "product_name": "string",
  "quantity": number,
  "unit": "string",
  "urgency": "low" | "medium" | "high"
}

Never add explanation outside the JSON object.
"""

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=256,
    system=SYSTEM,
    messages=[{"role": "user", "content": user_input}]
)

data = json.loads(response.content[0].text)  # safe to parse directly
For mission-critical structured output, use Claude's tool use feature (Module 3) instead. Tool use forces the model to output JSON matching your schema — it's more reliable than asking for JSON in the prompt.
