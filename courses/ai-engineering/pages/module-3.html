The fundamentals get you 80% of the way. These techniques close the gap for tasks that require multi-step reasoning, self-correction, or complex structured thinking — the kinds of problems where a naive prompt reliably fails.

3.1 Chain-of-Thought: Make the Model Reason Aloud

For tasks that require reasoning — math, logic, multi-step planning — asking the model to "think step by step" dramatically improves accuracy. The reasoning tokens aren't wasted: they shape every subsequent prediction.

# Without CoT — often wrong on multi-step problems:
"Q: A store has 120 apples. They sell 30%, then buy 50 more. How many do they have?"

# With CoT — add one instruction:
"""
Q: A store has 120 apples. They sell 30%, then buy 50 more. How many do they have?
Think step by step before giving your final answer.
"""

# Even better — structured CoT:
"""
Solve this math word problem. Use this structure:

Step 1: ...
Step 2: ...
Step 3: ...

N
"""
Claude's "extended thinking" mode (available via the API) does this internally with dedicated thinking tokens. Those tokens are processed but not always shown to users. For most prompting, explicit CoT in the prompt works well and is more controllable.
CoT adds output tokens, which adds cost and latency. Don't use it for classification, extraction, or any task where the answer is direct. Reserve it for problems where accuracy matters more than speed.
3.2 Prompt Chaining: Break Hard Tasks Into Steps

A single prompt trying to do five things at once produces mediocre results on all five. Prompt chaining runs multiple focused prompts in sequence, feeding each output as input to the next — like a pipeline.

import anthropic

client = anthropic.Anthropic()

def call(system, user):
    r = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system=system,
        messages=[{"role": "user", "content": user}]
    )
    return r.content[0].text

# Step 1: Extract key claims from a document
claims = call(
    "Extract the 3–5 most important factual claims from this text. One per line, no bullets.",
    document_text
)

# Step 2: Verify each claim against a knowledge source
verified = call(
    "For each claim below, assess: VERIFIED / UNVERIFIED / CONTRADICTED. One line per claim.",
    f"Claims to verify:\n{claims}\n\nSource document:\n{source_text}"
)

# Step 3: Write a summary that flags unverified claims
summary = call(
    "Write a 3-sentence summary. Mark any unverified claims with [UNVERIFIED].",
    f"Original text:\n{document_text}\n\nVerification results:\n{verified}"
)
💡 A chain of three focused 300-token prompts often beats one sprawling 900-token prompt. Each step gets the model's full attention on a narrow task, and you can inspect intermediate outputs to catch errors early.
3.3 Self-Critique and Reflection Loops

You can make the model review its own output by asking it to critique, then revise. This is especially useful for writing, code review, and any task where "good enough" is not good enough.

CRITIC_SYSTEM = """
You are a technical writing editor.
Review the draft below and list every issue you find.
Categories: [CLARITY] [ACCURACY] [COMPLETENESS] [TONE]
Be ruthless — your job is to find flaws, not validate.
"""

REVISER_SYSTEM = """
You are a technical writer.
Revise the draft to address every critique listed.
Return only the revised text with no explanation.
"""

# Generate initial draft
draft = call("Write a user-facing error message for a failed payment.", user_request)

# Critique it
critique = call(CRITIC_SYSTEM, f"Draft:\n{draft}")

# Revise based on critique
final = call(REVISER_SYSTEM, f"Original draft:\n{draft}\n\nCritique:\n{critique}")
3.4 Role Prompting and Persona Injection

Assigning the model a specific, credentialed role pulls relevant knowledge forward in its distribution. "You are a Python engineer" and "You are a security researcher" produce measurably different outputs for the same technical question.

# Generic role — weak:
"You are a helpful assistant."

# Specific role with stakes — much stronger:
"""
You are a senior security engineer who has done penetration testing
for Fortune 500 companies. You are paranoid by profession and assume
every system is compromised until proven otherwise.

When reviewing code, your first question is always: "How can this be abused?"
"""

# Domain-specific persona for a product:
"""
You are Maya, the AI assistant for Acme Corp's developer platform.
You only answer questions about Acme's APIs, SDKs, and documentation.
For anything outside that scope, say: "That's outside my area — check
docs.acme.com or contact support@acme.com."
You are friendly but concise. You never make up API endpoints.
"""
Don't confuse persona with expertise. "You are an expert" is weaker than describing specific expertise: "You have 15 years of experience writing production Rust and know every subtle unsafe invariant." The specificity of the credential shapes the response.
