Claude Code is Anthropic's agent framework for software engineering tasks. It gives Claude access to your filesystem, terminal, browser, and external tools — and it's extensible via hooks, skills, and MCP servers. This module shows you how to configure, extend, and build on top of it.

Claude API Setup

Before you can build agents with Claude Code, you need access to the Anthropic API. Claude Code uses the API under the hood, and you'll also need it for direct API calls in your own projects. Follow these steps to get your API key and set up your environment.

Important: The primary source for API keys is console.anthropic.com, not claude.ai. Claude.ai is the web interface; the API is managed separately.

Step 1: Get Your API Key

  1. Go to console.anthropic.com and sign in with your Anthropic account.
  2. Click API Keys in the left sidebar.
  3. Click Create Key, give it a descriptive name, and copy the generated key immediately (it won't be shown again).
  4. New accounts typically receive free trial credits (e.g., $5‑10 worth) to test the API. Check your balance in the Usage & Billing section.

Step 2: Install the Anthropic SDK

pip install anthropic

Step 3: Set Up Your Environment

Never hardcode your API key in your source code. Use environment variables:

# On macOS/Linux:
export ANTHROPIC_API_KEY='your-api-key-here'

# On Windows (PowerShell):
$env:ANTHROPIC_API_KEY='your-api-key-here'

# Or create a .env file:
ANTHROPIC_API_KEY=your-api-key-here
Security: Add .env to your .gitignore to prevent accidentally committing your key. Use a library like python-dotenv to load environment variables in Python.

Step 4: Verify Your Setup

Create a simple test script to confirm everything works:

import anthropic
import os

client = anthropic.Anthropic(
    api_key=os.environ.get("ANTHROPIC_API_KEY")
)

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=100,
    system="You are a helpful assistant.",
    messages=[
        {"role": "user", "content": "Say 'Hello, world!'"}
    ]
)

print(response.content[0].text)

Run the script. If you see Hello, world! (or a similar greeting), your API key is valid and the SDK is correctly installed.

💡 Personal discovery: I keep my API key in a password manager and load it via a shell script that sets the environment variable before starting my agent projects. This keeps it out of my shell history and makes it easy to rotate.

Checklist

  • I have created an API key at console.anthropic.com and copied it to a secure location
  • I have installed the Anthropic SDK via pip install anthropic
  • I have set the ANTHROPIC_API_KEY environment variable (or added it to a .env file)
  • I have run the verification script and received a successful response from Claude
5.1 What Claude Code Is (and How It Works)

Claude Code is a CLI agent that wraps the Claude API with a set of built-in tools: Read, Write, Edit, Bash, WebSearch, and more. It reads your project's CLAUDE.md for instructions, manages conversation context, and can spawn subagents for parallel work.

Unlike a bare API call, Claude Code maintains a persistent session with your project. It knows your git state, your file tree, and your recent changes. This context makes it far more capable at software tasks than the same model accessed directly. 
# Install Claude Code
npm install -g @anthropic-ai/claude-code

# Start a session in your project
cd your-project
claude

# Or run a one-shot command
claude "add error handling to src/api.py"

# Check version
claude --version

Under the hood, every Claude Code response is a ReAct loop: the model reasons, calls a tool (Read, Bash, Edit…), observes the result, and continues. Your permission settings control which tools run automatically and which require your confirmation.

5.2 CLAUDE.md — Your Agent's System Prompt

CLAUDE.md is automatically injected into the system prompt when Claude Code starts. It's the highest-leverage file in your project — the difference between an agent that immediately understands your codebase and one that wastes 20 tool calls orienting itself.

# CLAUDE.md — what to put in it

## Project Overview
One paragraph: what this project does, what stack it uses, who uses it.

## Architecture
Where the key files are. What not to touch. Module boundaries.

## Conventions
Naming conventions, code style, testing approach, commit format.

## Common Tasks
How to run tests. How to start the dev server. How to deploy.
Include the exact commands — don't make the agent guess.

## What NOT to Do
Explicit prohibitions save you from expensive mistakes:
- Never edit generated files in build/
- Never use `git push --force` on main
- Never add external dependencies without asking first
A CLAUDE.md that's too long is as bad as no CLAUDE.md — the agent gets lost in it. Keep it under 500 lines. Write it as if briefing a new senior engineer on their first day: what they absolutely need to know, nothing more.
5.3 Hooks: Automate Behaviors Around Tool Calls

Hooks are shell commands that Claude Code runs automatically before or after specific events — before a tool call, after a tool call, on session start, on session stop. They're how you enforce invariants without relying on the model to remember.

# .claude/settings.json — hook configuration
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'About to run bash command' >> /tmp/claude-audit.log"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "cd $PROJECT_ROOT && npm run lint --silent 2>&1 | head -20"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "cd $PROJECT_ROOT && npm test --silent 2>&1 | tail -5"
          }
        ]
      }
    ]
  }
}
💡 The hook setup that changed how I use Claude Code: a PostToolUse hook on Write that runs the linter immediately after every file write. The agent sees lint errors in real-time and fixes them before moving on — instead of writing 5 files and then discovering they all have issues.
5.4 Skills and MCP Servers: Extend the Agent

Skills are reusable prompt templates stored in .claude/skills/ — invoke them by name to run complex, multi-step tasks without re-explaining them every time. MCP (Model Context Protocol) servers add new tools and data sources to the agent.

# .claude/skills/create-component/SKILL.md
# Scaffold a new React component with tests and story

When this skill is invoked:
1. Ask for: component name, props, whether it needs a story
2. Create src/components/{Name}/{Name}.tsx
3. Create src/components/{Name}/{Name}.test.tsx with basic render test
4. Create src/components/{Name}/{Name}.stories.tsx if requested
5. Export from src/components/index.ts
6. Run: npm test -- --testPathPattern={Name} to verify

Always follow the project's TypeScript strict mode. Never use `any`.

---

# Register an MCP server for GitHub access
# .claude/settings.json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
5.5 Subagents and Parallel Work

Claude Code can spawn subagents — independent agent instances with their own context window — to parallelize work. An orchestrator agent can delegate "research X" and "research Y" to two subagents simultaneously, then synthesize both results.

# The Agent tool: spawn a subagent from within an agent
# (this is what Claude Code itself does internally)

# From the orchestrator's perspective:
Agent({
  subagent_type: "general-purpose",
  prompt: "Read all files in src/api/ and list every endpoint with its HTTP method, path, and description. Return as JSON array.",
  run_in_background: false
})

# The subagent gets its own context window and tools.
# Results are returned to the orchestrator as a message.
# The orchestrator can spawn multiple subagents in parallel
# by calling Agent() multiple times in one response.
Each subagent starts cold — it doesn't inherit the parent's conversation history. Brief each subagent like a new hire: give it exactly what it needs to know, no more. Long subagent prompts are expensive and often counterproductive.
5.6 Verify: Configure Your Project for Claude Code

The best way to learn Claude Code is to configure it for a real project you already work on. Set up CLAUDE.md, add one hook, and write one skill for a task you do repeatedly.

  • I installed Claude Code and ran a basic task in a real project
  • I wrote a CLAUDE.md that covers: overview, architecture, conventions, and common commands
  • I added at least one hook (PreToolUse or PostToolUse) that enforces a project invariant
  • I created a skill in .claude/skills/ for a task I repeat more than once a week
  • I connected at least one MCP server (GitHub, filesystem, or a custom one)
