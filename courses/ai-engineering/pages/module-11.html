Context engineering is the discipline of deciding what information goes into the model's context window — and what stays out. Most agent failures are not model failures. They're context failures: too much noise, the wrong information at the wrong time, or no awareness of cost. This module gives you the mental model and the mechanics.

8.1 What Is Context Engineering (and Why Agents Fail Without It)

Every API call you make includes a context window — the full list of messages the model sees at inference time. The model cannot learn between calls. It cannot remember yesterday. It can only reason about what you put in front of it right now.

This means your job as an agent builder is not just "write a good prompt." It's to engineer — deliberately design — exactly what the model sees at each step.

More context is not better. Every token you add competes with signal for the model's attention. Dumping a 50,000-token codebase into context doesn't make the model smarter — it makes it slower, more expensive, and often less accurate. Relevant context wins.

Three root causes of agent context failures:

# 1. Noise overload — too much irrelevant information
# The model hallucinates or ignores the right facts because
# they're buried in 10,000 tokens of tangential content.

# 2. Missing prerequisites — the model lacks key facts
# You asked the agent to file a bug, but it doesn't know
# your project's Jira URL or issue template format.

# 3. Stale context — information that was true earlier but isn't now
# A multi-step agent appends every tool result to context.
# By step 15, the step-1 results are obsolete and contradictory.

# The fix for all three: treat context as a living artifact,
# not an ever-growing log. Curate it at every step.
The context window is not a transcript. Think of it as the whiteboard in front of the model — you decide what's written on it. A good context engineer is constantly erasing the irrelevant and surfacing the essential.
8.2 Structuring Your System Prompt

The system prompt is the highest-priority context — it's always there, it frames everything else, and the model weights it most heavily. A well-structured system prompt answers four questions in order:

SYSTEM PROMPT TEMPLATE
======================

## Identity
Who you are, what you're for, who your user is.
"You are a code review assistant for a Python backend team at a fintech startup."

## Capabilities
What you can and cannot do. Be explicit about limitations.
"You can read files, run tests, and suggest changes. You cannot push to git."

## Constraints
Rules that override everything else.
"Never suggest deleting a migration file. Always explain the 'why' of a refactor.
Prefer explicit over implicit. Follow PEP 8 unless the file already diverges."

## Output format
Exactly how the response should be shaped.
"When reviewing code: (1) start with a one-line verdict, (2) list issues as
bullet points with severity [HIGH/MED/LOW], (3) end with a concrete next step."
Ordering matters inside the system prompt. Information at the top of the system prompt gets more weight than information at the bottom — especially in long prompts. Put your most critical constraints first, not last.

What goes where in the message array:

messages = [
    # System prompt — identity, capabilities, constraints, format
    # (passed separately in the API, not as a message role)

    # User messages — the actual task, with just enough context
    {"role": "user", "content": "Review the diff in context."},

    # Assistant messages — prior reasoning (only include if needed)
    # {"role": "assistant", "content": "..."},

    # Tool results — factual outputs, not prose
    # Include only results relevant to the current step
]
💡 What worked for me: separating "persona" from "constraints" in the system prompt. I had one section that described what the agent is (identity + capabilities) and a separate section — formatted as a numbered list — for hard constraints. The numbered list format signals to the model that these are enumerable, checkable rules, not soft suggestions. Response quality improved noticeably.
8.3 Dynamic Context Assembly

Static prompts are a starting point. Real agents assemble context dynamically — pulling in only the facts that matter for this specific request, at this moment.

import anthropic

client = anthropic.Anthropic()

def build_context(user_query: str, knowledge_base: list[dict]) -> str:
    """Return the top-3 most relevant knowledge base entries as context."""
    # In production, use embeddings + vector similarity.
    # Here we demo keyword overlap as a simple proxy.
    scored = []
    query_words = set(user_query.lower().split())
    for entry in knowledge_base:
        words = set(entry["text"].lower().split())
        score = len(query_words & words)
        scored.append((score, entry))

    top = sorted(scored, key=lambda x: x[0], reverse=True)[:3]
    return "\n\n".join(f"[Source: {e['id']}]\n{e['text']}" for _, e in top)


def answer_with_context(user_query: str, knowledge_base: list[dict]) -> str:
    relevant_context = build_context(user_query, knowledge_base)

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="You are a helpful assistant. Answer using only the provided context.",
        messages=[
            {
                "role": "user",
                "content": f"Context:\n{relevant_context}\n\nQuestion: {user_query}"
            }
        ]
    )
    return response.content[0].text

When context grows too large — a long agent loop, a large codebase — you need a truncation strategy. Choose one based on your task:

# Strategy 1: Sliding window — keep the N most recent messages
def sliding_window(messages: list, max_messages: int) -> list:
    # Always keep the first message (usually the task description)
    if len(messages) <= max_messages:
        return messages
    return [messages[0]] + messages[-(max_messages - 1):]


# Strategy 2: Summarize-and-compress
# When the window is full, call Claude to summarize earlier turns,
# replace them with the summary, then continue.
def compress_history(messages: list, client) -> list:
    if len(messages) < 10:
        return messages
    to_compress = messages[1:-4]  # keep first + last 4
    summary_response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=512,
        messages=[{
            "role": "user",
            "content": f"Summarize the key facts and decisions from this conversation:\n{to_compress}"
        }]
    )
    summary = summary_response.content[0].text
    return [messages[0], {"role": "assistant", "content": f"[Summary of earlier work]: {summary}"}] + messages[-4:]
Use a cheap, fast model (Haiku) for summarization. You're compressing facts, not reasoning — the small model handles it well, and the cost is a fraction of what you'd spend sending the full history to Sonnet or Opus.
8.4 Prompt Caching for Speed and Cost

Every time you send the same prefix to the Claude API — the same system prompt, the same conversation history — you pay to process those tokens again. Prompt caching lets Claude reuse the KV-cache from a previous call. For agents with large, stable system prompts, this cuts latency by ~80% and cost by ~90% on cached tokens.

Caching works on the longest common prefix. The rule: put your stable content first (system prompt, reference docs, tool definitions), and your dynamic, per-request content last. The stable prefix is cached; only the new suffix is computed fresh. 
import anthropic

client = anthropic.Anthropic()

# Your large, stable system prompt — loaded once, cached across calls
SYSTEM_INSTRUCTIONS = open("system_instructions.txt").read()  # e.g. 10,000 tokens

def ask_with_caching(user_message: str) -> str:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system=[
            {
                "type": "text",
                "text": SYSTEM_INSTRUCTIONS,
                "cache_control": {"type": "ephemeral"}  # ← cache this prefix
            }
        ],
        messages=[
            {"role": "user", "content": user_message}
        ]
    )

    # Inspect cache usage in the response
    usage = response.usage
    print(f"Cache read:  {usage.cache_read_input_tokens} tokens")
    print(f"Cache write: {usage.cache_creation_input_tokens} tokens")
    print(f"Not cached:  {usage.input_tokens} tokens")

    return response.content[0].text


# First call: cache miss — all tokens billed at full rate
result1 = ask_with_caching("What are the project conventions?")

# Second call: cache hit — 90% cost reduction on the system prompt
result2 = ask_with_caching("How do I run the tests?")
The cache TTL is 5 minutes by default. If your agent pauses for more than 5 minutes between calls — waiting for human input, a long tool execution — the cache expires and you pay full price again. Design your agent loop to call the API at least every few minutes if you need sustained caching benefits.
8.5 Verify: Audit Your Context Quality

The fastest way to improve an agent is to print its context at the moment it makes a bad decision, then ask: "What is wrong with what the model sees here?" You will find the answer almost every time.

  • My system prompt has four clearly separated sections: identity, capabilities, constraints, and output format
  • My most critical constraints appear at the top of the system prompt, not the bottom
  • I assemble context dynamically — I only inject the information relevant to the current query, not everything I have
  • I have a truncation or compression strategy for long agent loops (sliding window or summarize-and-compress)
  • I implemented prompt caching with cache_control on my stable system prompt prefix
  • I can print the full context my agent sees at any step, and I've used it to debug at least one bad output
