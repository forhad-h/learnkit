A prompt answers one question. An agent solves a problem. The difference is a loop: the model decides what to do, takes an action, observes the result, and decides again. This section gives you the architecture vocabulary and patterns you'll implement in Sections 8 and 9.

7.1 The ReAct Loop: Reason → Act → Observe

The ReAct pattern (Reasoning + Acting) is the foundation of most production agents. The model alternates between reasoning about the current situation and taking an action, with each action's result fed back as an observation for the next reasoning step.

# ReAct loop pseudocode:
#
# Thought:  I need to find the current weather in Tokyo.
# Action:   call_tool("get_weather", {"city": "Tokyo"})
# Observation: {"temp": 18, "condition": "cloudy", "humidity": 72}
#
# Thought:  I have the weather. Now I need to check if it will rain.
# Action:   call_tool("get_forecast", {"city": "Tokyo", "hours": 24})
# Observation: {"rain_probability": 0.85, "rain_start": "14:00"}
#
# Thought:  I have enough information to answer.
# Answer:   "It's 18°C and cloudy in Tokyo right now. Rain is expected
#            at 2pm with 85% probability — bring an umbrella."

# This loop continues until the agent emits a final "Answer" instead of an "Action"
The model doesn't run the loop — your code does. The model produces a thought + action text block. Your code parses the action, executes it, appends the observation to the message history, and calls the model again. This is the "agentic loop."
7.2 Memory: Short-term, Long-term, and Semantic

Agents need memory beyond the context window to handle long tasks and learn across sessions. There are three distinct memory types, and production agents use all three.

TypeStorageLifetimeUse case
In-context (short-term)Message historyOne sessionMulti-turn conversation, task steps
External (long-term)Database / filesPermanentUser preferences, completed work, facts
Semantic (episodic)Vector storePersistent + searchable"What did we discuss about X last month?"
 
# Pattern: write key decisions to long-term memory mid-task
def run_agent_step(task, history, memory):
    response = model.call(
        system=AGENT_SYSTEM,
        messages=history + [{"role": "user", "content": task}]
    )

    # If the model produced a memory-write action, persist it
    if response.action == "remember":
        memory.save(response.key, response.value)  # durable storage

    # Inject relevant long-term memories into next context
    relevant = memory.search(task)
    history.append({"role": "user", "content": f"[Memory]\n{relevant}\n\n{task}"})

    return response
7.3 Multi-Agent Patterns: Orchestrator + Subagents

For complex tasks, a single agent becomes a bottleneck. Multi-agent architectures split work: an orchestrator breaks the task into subtasks and delegates to specialized subagents. Each subagent has its own system prompt, tools, and context window.

# Orchestrator pattern:
#
#          User request
#               ↓
#        [Orchestrator Agent]
#        "Research + write report"
#               ↓
#     ┌─────────┴──────────┐
#     ↓                    ↓
# [Research Agent]   [Writer Agent]
# tools: web_search  tools: read_file, write_file
# returns: raw notes returns: formatted report
#               ↓
#        [Orchestrator Agent]
#        "Combine, verify, deliver"
#               ↓
#           Final report

# In code: the orchestrator calls subagents as tools
def research_and_write(topic: str) -> str:
    notes = research_agent.run(f"Research: {topic}")
    report = writer_agent.run(f"Write a report based on these notes:\n{notes}")
    return orchestrator_agent.run(f"Review and finalize:\n{report}")
Multi-agent systems multiply both capability and complexity. Errors compound — a bad output from a subagent can derail the entire pipeline. Add validation checkpoints between agents and log every intermediate result. Debugging a silent failure in a 5-agent chain is painful.
7.4 When to Use Agents (and When Not To)

Agents are powerful but expensive and hard to debug. A simple prompt chain is faster, cheaper, and more predictable for most tasks. Use an agent only when you genuinely can't know the steps in advance.

Use a prompt chain when...Use an agent when...
The steps are known and fixedThe steps depend on intermediate results
The task completes in 1–5 LLM callsThe task requires dynamic tool selection
You need deterministic, auditable behaviorThe task requires exploration or planning
Latency is critical (<2s)Quality matters more than speed
