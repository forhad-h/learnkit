What goes into the context window determines the quality of every response. Context engineering is the discipline of deciding what to include, what to exclude, and in what order — because the model's attention is finite and its behavior is shaped by everything it sees.

4.1 What Belongs in the Context Window

Every token in your context window costs money and competes for attention. Fill it deliberately. The model doesn't benefit from reading irrelevant content — it's distracted by it.

IncludeExclude
The specific document, code, or data relevant to this taskEntire codebases, full database dumps
The last 3–5 messages of conversation historyFull conversation history back to session start
Examples that match the current task typeExamples for task types not being requested
Relevant retrieved chunks (from RAG)All retrieved results including low-relevance ones
Constraints and format instructionsLengthy explanations of why constraints exist
Research shows model performance degrades when the relevant information is buried in the middle of a long context ("lost in the middle" problem). Put the most important content at the beginning (system prompt) or end (most recent user message), not in the middle.
4.2 Conversation History Management

Long conversations eat your context budget fast. You need a strategy for managing history — which messages to keep, which to summarize, and which to drop.

def build_context(messages: list[dict], max_tokens=4000) -> list[dict]:
    """Keep recent messages; summarize older ones when budget is tight."""
    recent = messages[-6:]  # always keep last 3 turns (6 messages)
    older = messages[:-6]

    if not older:
        return recent

    # Summarize the older history into a single system message
    summary_prompt = "Summarize this conversation history in 3 bullet points, preserving key decisions and facts."
    summary = call(summary_prompt, format_messages(older))

    return [
        {"role": "user", "content": f"[Earlier conversation summary]\n{summary}"},
        {"role": "assistant", "content": "Understood, I have the context from our earlier discussion."},
        *recent
    ]
Naively truncating old messages loses important context ("we already decided X three turns ago"). Summarization preserves the decisions while cutting the token cost. For production apps, store the full history in a database — only send the condensed version to the API.
4.3 Prompt Caching for Cost and Speed

Claude's prompt caching lets you pay up to 90% less for repeated content. Any prefix you send identically in multiple requests gets cached — reads cost 10% of a fresh token, writes cost 25% more than normal but only happen once every 5 minutes.

import anthropic

client = anthropic.Anthropic()

# Mark stable content for caching with cache_control
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": LARGE_SYSTEM_PROMPT,  # 5,000+ tokens of stable instructions
            "cache_control": {"type": "ephemeral"}  # cache this prefix
        }
    ],
    messages=[
        {
            "role": "user",
            "content": [
                {
                    "type": "text",
                    "text": LARGE_DOCUMENT,  # document being analyzed
                    "cache_control": {"type": "ephemeral"}
                },
                {
                    "type": "text",
                    "text": user_question  # dynamic — not cached
                }
            ]
        }
    ]
)

# Check cache performance in response headers
usage = response.usage
print(f"Cache read tokens: {usage.cache_read_input_tokens}")
print(f"Cache write tokens: {usage.cache_creation_input_tokens}")
💡 The pattern that cut my costs by 70%: cache the system prompt + any large document that's fixed for a session (like a codebase or report). Only the user's question is dynamic. Every follow-up question in that session reads the cache.
